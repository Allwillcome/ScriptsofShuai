# 一位产品经理的AI对话历险记

——从想要放弃到掌握心法的40分钟

---

我是一名AI产品经理，日常和各类AI工具打交道。

最近负责一个健身数据产品，其中有个「肌肉负荷页面」需要设计规则式AI评语，作为用户点击AI助手的入口。页面是个人体热力图，标注全身20个肌肉群的负荷状态——高负荷、低负荷、无负荷。

我需要根据这些数据，让AI生成有价值的评语。

## 开始

我把原型图发给Claude Code，提了需求：

> "根据这个页面撰写规则式的AI评语，作为AI对话的入口。"

心想这需求够清楚了吧？

Claude Code很积极，立刻进入Plan模式开始分析。然后给了我一个庞大的方案：17个场景、负荷等级判定、恢复进度判定、肌肉不平衡检测、TypeScript接口、评语生成器逻辑......

我看着那满满当当的文档，第一反应是：**这什么玩意儿？**

感觉它在"吹牛皮"，给了一堆看似专业但完全用不上的东西。我甚至开始怀疑：是不是模型的问题？要不要换个模型试试？比如切换到Sonnet，或者用MiniMax？

那一刻我真的想放弃了。

## 转机

但本着"来都来了"的心态，我耐住性子，又补了一段话：

> "你不需要调研使用什么技术框架，我的目的是基于这个页面的内容，来设想用户有哪些场景，根据这些场景来生成AI评语。文字评语就行，场景要足够全面、描述要足够简洁。"

我还专门列出了肌肉群的完整列表：

> "PeakWatch的肌肉群是这些：上臀、下臀、腹部、上胸、中下胸、上背、下背，股四头肌、腘绳肌、小腿，肩前束、中束、后束、斜方肌、二头、三头、前臂"

这是第一次意识到：**AI需要我提供足够的上下文。**

## 继续调整

然而沟通还是有问题——

AI还是习惯性地设计复杂方案，而且总喜欢用"今天""本次"这样的时间限定词。但我这个页面支持7天、30天、90天、1年切换的，不能用具体日期啊。

我又继续补：

> "去掉'今天'的描述，改成更通用的说法。另外，无数据的场景就直接用'数据累计后将提供建议'。"

AI终于听懂了。

## 结果

第3轮、第4轮沟通后，AI逐渐理解了我的真实需求。

最终输出变成了简洁的6个核心场景：

| 场景 | 触发条件 |
|------|----------|
| 场景1 | 大肌肉群有低负荷 → 提醒练低负荷部位 |
| 场景2 | 所有大肌肉群都练了 → 推荐练腹部 |
| 场景3 | 无训练数据 → 新手引导 |
| 场景4 | 全部无负荷 → 数据累计后提供建议 |
| 场景5 | 仅小肌肉群高负荷 → 建议优先大肌群 |
| 场景6 | 大肌群均衡 → 保持鼓励 |

从17个场景精简到6个，每个都能直接落地。这才是我真正想要的。

## 数据

沟通结束后算了下：

- 实际对话时间：约40分钟
- 对话轮数：9轮
- 中途差点放弃：2次

## 心得

总结下来就几点：

> **降低预期，增加耐心，和AI协同工作，不当甩手掌柜。**

具体技巧：

1. 补充具体上下文——肌肉群列表、负荷有几种、页面支持哪些时间维度
2. 明确输出格式——"超过两行"、"用Markdown表格"
3. 及时纠正方向——哪里不对说哪里，别让AI猜
4. 善用Plan模式——先把需求聊清楚，再切回常规模式快速输出

## 感悟

回头看这段经历，最大的收获是：

**AI确实适合帮助从0到1实现内容，但前提是你脑子里要先有大概的方向和架构。**

AI不是万能的甩手掌柜——你得把需求说清楚，把上下文补齐，它才能帮你完善。

而且使用AI过程中遇到很多不爽，那恰恰是成长的信号。

如果不是亲自用了这么多次，我也不可能沉淀出这么多经验。

遇到困难别急着放弃，补充上下文，调整预期，耐住性子——AI真的能帮上忙。
