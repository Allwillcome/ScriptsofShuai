# ProductSolutionOpt Skill

产品方案优化思路，聚焦于用最小改动解决产品问题，同时保证用户体验的稳定性。

## 核心思考角度

### 1. 最小改动原则
- 优先考虑通过**配置层面**解决问题，而非大规模重构
- 判断是后端改动还是前端改动，还是两者都需要
- 明确具体的改动点，用表格形式输出

### 2. 保持 UI 稳定性
- **选项置灰优于隐藏**：当某个选项在特定条件下不可用时，保留选项但置灰（disabled），而不是直接隐藏
- 原因：用户切换设备/场景时，界面元素的突然变化会造成困惑，置灰能保持界面稳定性
- 配合辅助文字说明为何不可用

### 3. 前置条件检测
- 先明确定义**前置条件**（什么情况下触发特殊处理）
- 明确检测方式（设备类型、连接方式、数据状态等）
- 区分不同场景下的行为差异

### 4. 辅助信息设计
- 当选项不可用时，提供清晰的辅助文字说明原因
- 文案要精简、准确
- 避免技术术语，用用户能理解的语言

## 典型问题模式

| 场景 | 推荐方案 |
|------|----------|
| 多设备数据源兼容 | 选项置灰 + 辅助文字说明 |
| 切换设备时的配置继承 | 保持选项可见，自动回退到可用选项 |
| 功能在特定条件下不可用 | 置灰 + toast 提示或辅助文字 |

## 输出格式

当分析产品优化方案时，输出以下结构：

```
## 方案分析

### 核心思路
{描述问题的本质和解决方向}

### 改动点
| 层级 | 改什么 |
|------|--------|
| {后端/前端/配置} | {具体改动内容} |

### 前置条件
{什么情况下触发}

### 辅助文字（如适用）
{当选项不可用时显示的文字}
```

## 使用场景

当用户讨论以下问题时使用：
- "这个功能在某些设备上不能用，怎么处理"
- "用户切换设备时体验不好，怎么优化"
- "能不能用最小的改动解决这个问题"
- "给一个方案"
- "考虑下用户体验"

## 示例

**问题**：Apple Watch 和 Garmin 都接入了 app，但 HRV 来源配置中"苹果健康"选项只对 Apple Watch 有效，Garmin 用户看到这个选项会困惑。

**分析思路**：
1. 最小改动：不在后端过滤选项，而是前端检测设备类型后置灰
2. UI 稳定性：保留选项但置灰，用户切换设备时界面不变
3. 前置条件：检测到通过云端连接了佳明设备
4. 辅助文字：说明为什么不可用

**输出**：
- 前置条件：通过云端连接了佳明设备
- 苹果健康选项：置灰（disabled）
- 辅助文字：PeakWatch 使用原始心率计算 RMSSD；苹果健康方式仅 Apple Watch 可用，连接佳明时无法使用
